Setup & operations guide

Link once with the UI. Run Signal headless afterward.

This guide takes you from an empty Docker volume to a linked Signal Desktop profile, a Node-only daemon, incoming webhooks, outgoing messages, and unencrypted machine-to-machine state transfer through Cloudflare R2.

Docker Compose Linux amd64 Node-only daemon Unencrypted R2 snapshots Direct text messages 90-day builds

Read this first

Never run two copies of the same linked-device profile. Stop the source machine completely before snapshotting it, and do not start the destination until restore has finished. A local lock protects containers on one host; it cannot coordinate two hosts.
Current scope: direct one-to-one text receive and send, quoted replies, native outbound text formatting, and outbound emoji reactions to locally retained messages. Groups, attachments, previews, incoming reactions, reaction removal, deletes, stories, calls, sync transcripts, and disappearing-message delivery are not implemented. Unsupported content does not produce webhooks. Unsupported ciphertext payloads are retained in encrypted staging; empty unsupported control envelopes are acknowledged without staging because they contain no payload to preserve.
A real account link and live Signal send/receive acceptance test still needs to be performed by you. Automated tests do not register or authenticate an account. The authenticated headless transport probes /v1/keepalive every 30 seconds and automatically reconnects stale sockets.

Three containers, one volume

signal-ui links or displays the profile, signal runs headless, and state moves an offline profile through R2. They all use signal-desktop-data by default.

Safe default networking

The noVNC UI and HTTP API publish only on 127.0.0.1. Use an SSH tunnel for remote access; do not expose either port directly to the public internet.

1. Prerequisites

  • Docker Engine and Docker Compose v2
  • A Signal account in the Android or iOS mobile app
  • An amd64 host, or Linux amd64 emulation
  • Enough memory and disk to compile Signal Desktop initially
  • For remote transfer: a private R2 bucket and S3 API credentials

Clone the repository and enter it:

git clone YOUR_FORK_URL signal-desktop-cli
cd signal-desktop-cli

2. Create your configuration

Copy the sanitized template. The resulting .env file is ignored by Git, but it still contains secrets and should be mode 0600. 

cp .env.example .env
chmod 600 .env

Edit .env and replace every placeholder:

SIGNAL_UI_PASSWORD=vncpass
SIGNAL_API_TOKEN=replace-with-at-least-16-random-characters
SIGNAL_WEBHOOK_URL=https://example.com/signal/webhook
SIGNAL_WEBHOOK_SECRET=replace-with-at-least-16-random-characters
SENTRY_DSN=https://public-key@organization.ingest.sentry.io/project-id

R2_ACCOUNT_ID=your-cloudflare-account-id
R2_BUCKET=your-private-bucket
R2_ACCESS_KEY_ID=your-r2-access-key-id
R2_SECRET_ACCESS_KEY=your-r2-secret-access-key
R2_PREFIX=signal-state
VNC limitation: the UI password must contain at most eight characters. This is an RFB protocol limit. Host-loopback binding and SSH tunnelling are part of the security boundary.
R2 snapshots are intentionally unencrypted. Anyone who can read a snapshot can access the complete portable Signal profile, including the material needed to open its database. Use a private bucket, narrowly scoped credentials, and appropriate R2 access controls. Checksums detect ordinary corruption but do not authenticate data against an attacker with R2 write access.

3. Link your Signal account

  1. Start only the GUI container.
  2. Open the local noVNC page and enter your UI password.
  3. Use the Signal mobile app to scan the linked-device QR code.
  4. Wait for Signal Desktop to finish linking and initialization.
  5. Stop the UI cleanly before starting the daemon or state tool.

Start the UI

docker compose --profile ui up --build signal-ui

Open http://127.0.0.1:6080/. On a remote host, create a tunnel first: 

ssh -L 6080:127.0.0.1:6080 user@remote-host

Scan the QR code

In the mobile Signal app, open Settings → Linked devices, choose Link new device, and scan the QR code shown inside the containerized Signal Desktop window.

Stop the UI safely

docker compose --profile ui stop --timeout 150 signal-ui
docker compose ps

Confirm that neither signal-ui nor signal is running. Never use docker compose down -v unless you intend to delete the linked profile permanently.

4A. Run headless on the same machine

If the daemon will use the volume where you linked the account, R2 is not required. Start the daemon after the UI has stopped: 

docker compose up --build -d signal
docker compose logs -f signal

Check process health and Signal readiness

curl --fail http://127.0.0.1:8080/healthz
curl --fail http://127.0.0.1:8080/readyz
  • /healthz means the local HTTP process is alive.
  • /readyz returns 200 only when the profile, database, build, and online Signal transport are ready.
  • A 503 response includes a JSON status and reason for diagnosis.

4B. Move the linked profile through R2

Both the source UI and source daemon must be stopped before push. Keep the destination stopped until pull completes.

Upload from the source machine

docker compose stop --timeout 45 signal
docker compose --profile ui stop --timeout 150 signal-ui
docker compose --profile tools run --rm --build state push

Record the printed snapshot UUID. The tool archives the complete meaningful Signal profile into an unencrypted .tar.zst archive, uploads an immutable object, and publishes metadata last. Upload is not tied to the original linking machine: whichever stopped machine currently owns the profile can push it.

List and verify

docker compose --profile tools run --rm state list
docker compose --profile tools run --rm state verify latest

Verification downloads the snapshot, checks size and SHA-256, decompresses it in temporary storage, validates safe archive paths, and compares all files with the included manifest. It does not activate the snapshot.

This is snapshot format 2. Earlier format-1 snapshots are ignored; create a new snapshot before relying on restore.

Restore on the destination

Clone the same repository revision, create the destination .env with the same R2 configuration, and run restore before starting the daemon: 

docker compose --profile tools run --rm --build state pull latest
docker compose up --build -d signal
curl --fail http://127.0.0.1:8080/readyz

The state command creates the named Docker volume automatically. A preliminary empty daemon start is unnecessary. To replace a non-empty destination profile, use a specific verified UUID: 

docker compose --profile tools run --rm state pull SNAPSHOT_UUID --replace

Replacement is staged and verified first, then atomically exchanged on Linux. The volume filesystem must support renameat2(RENAME_EXCHANGE).

5. HTTP API reference

The control API is available at http://127.0.0.1:${SIGNAL_API_PORT:-8080} on the Docker host. Compose publishes it only to loopback. Use the service name and a private shared Compose network when calling it from another container; do not expose it publicly without a separate TLS/authentication gateway and network controls.

All successful and error responses are JSON with Content-Type: application/json; charset=utf-8. GET /healthz and GET /readyz do not require authentication. Every POST /v1/... request requires Authorization: Bearer ${SIGNAL_API_TOKEN} and Content-Type: application/json; request bodies are limited to 64 KiB.

Method Path Authentication Purpose
GET /healthz No Check that the local control process is running.
GET /readyz No Check whether the profile and Signal runtime can send.
POST /v1/messages Bearer token Send a direct text message.
POST /v1/reactions Bearer token Add or replace this account's reaction on a retained direct message.

GET /healthz

Use this liveness endpoint to determine whether the local control process is running. It has no request body. 

curl --fail http://127.0.0.1:8080/healthz
{ "status": "ok" }

GET /readyz

Use this readiness endpoint before sending. It has no request body and returns HTTP 200 only when ready is true; otherwise it returns HTTP 503 with the same response shape. reason is optional and is present when the daemon has a diagnostic reason for not being ready. 

curl --fail-with-body http://127.0.0.1:8080/readyz
{
  "buildExpiration": {
    "createdAt": 1783960000000,
    "createdAtIso": "2026-07-13T16:26:40.000Z",
    "daysRemaining": 89.5,
    "expired": false,
    "expiresAt": 1791736000000,
    "expiresAtIso": "2026-10-11T16:26:40.000Z",
    "validityDays": 90
  },
  "connected": true,
  "databaseReady": true,
  "linked": true,
  "phase": "ready",
  "ready": true
}

POST /v1/messages

Send one direct text message. The request object is strict: fields not listed below, including idempotency_key, are rejected. Each accepted request creates a new outgoing message ID, so retrying an ambiguous request can send a duplicate.

Field Required Type Rules
destination Yes string An international E164 number (+ followed by 7–15 digits) already mapped in the restored profile, or a lowercase Signal ACI.
body Yes string Text to send, 1–32,768 characters.
parse_mode No string Exact value Markdown. Enables **bold**, _italic_, ~~strikethrough~~, `monospace`, and ||spoiler||; without it, body is sent unchanged.
quote_message_id No string Opaque message.message_id from an earlier webhook. The message must still exist locally and belong to the destination conversation. 
curl --fail-with-body \
  -H "Authorization: Bearer ${SIGNAL_API_TOKEN}" \
  -H 'Content-Type: application/json' \
  --data '{
    "destination": "+12025550123",
    "body": "**hello** from the headless client",
    "parse_mode": "Markdown",
    "quote_message_id": "incoming-message-uuid"
  }' \
  http://127.0.0.1:8080/v1/messages
{
  "destination": "00000000-0000-0000-0000-000000000000",
  "messageId": "00000000-0000-0000-0000-000000000000",
  "status": "sent",
  "timestamp": 1783960000000
}

Formatting markers are removed from the sent text. Formatting can be nested, a backslash escapes the next character, and unmatched markers remain literal. Signal may repair a recipient device-list mismatch and retransmit the same request once after Signal explicitly rejects the first encrypted payload; other transport failures are not automatically retried.

POST /v1/reactions

Add one supported emoji reaction to a locally retained direct message, or replace this account's existing reaction on that message. Reaction removal is not exposed. The request object is strict.

Field Required Type Rules
destination Yes string E164 number already mapped in the restored profile, or a lowercase Signal ACI.
message_id Yes string Opaque message.message_id from an earlier webhook. The target must still exist locally and belong to the destination conversation.
emoji Yes string One Signal-supported emoji. 
curl --fail-with-body \
  -H "Authorization: Bearer ${SIGNAL_API_TOKEN}" \
  -H 'Content-Type: application/json' \
  --data '{
    "destination": "+12025550123",
    "message_id": "incoming-message-id",
    "emoji": "👍"
  }' \
  http://127.0.0.1:8080/v1/reactions
{
  "destination": "00000000-0000-0000-0000-000000000000",
  "emoji": "👍",
  "messageId": "incoming-message-id",
  "status": "sent",
  "timestamp": 1783960000000
}

Error responses

The API returns this error object for failed sends and unknown routes: 

{
  "error": {
    "code": "not-connected",
    "message": "Signal service is not connected",
    "retryable": true
  }
}
Status Error code Meaning
400 invalid-request The JSON, content type, body size, or strict request validation failed.
401 unauthorized Bearer authentication is missing or invalid.
404 not-found / recipient-not-found The endpoint does not exist, or an E164 destination has no local ACI mapping.
409 identity / send-failed Signal rejected delivery permanently or a safety-number identity condition needs attention.
422 unsupported The requested Signal content is unsupported.
429 rate-limited Signal temporarily limited the request.
502 network / device-mismatch Temporary Signal delivery failure; retry only when retryable is true.
503 not-connected The daemon is not ready or connected to Signal.

6. Send a direct text message

The message endpoints POST /v1/messages and POST /v1/reactions require authentication. Health endpoints do not require authentication. When SIGNAL_DAEMON_CONNECT=true, startup requires SIGNAL_API_TOKEN with at least 16 characters.

Compose reads .env for containers, but does not export variables into your shell. Export the token or substitute its value in the request without saving it in shell history. 

export SIGNAL_API_TOKEN='your-configured-token'

curl --fail-with-body \
  -H "Authorization: Bearer ${SIGNAL_API_TOKEN}" \
  -H 'Content-Type: application/json' \
  --data '{
    "destination": "+12025550123",
    "body": "**hello** from the headless client",
    "parse_mode": "Markdown",
    "quote_message_id": "incoming-message-uuid"
  }' \
  http://127.0.0.1:8080/v1/messages

Successful response:

{
  "destination": "00000000-0000-0000-0000-000000000000",
  "messageId": "00000000-0000-0000-0000-000000000000",
  "status": "sent",
  "timestamp": 1783960000000
}
  • Destination: lowercase Signal ACI, or E164 already mapped in the restored profile.
  • Body: 1–32,768 characters.
  • Optional quoted reply: set quote_message_id to the opaque ID from an earlier webhook's message.message_id. The referenced message must still exist locally and belong to the destination conversation.
  • Optional formatting: set parse_mode to the exact value Markdown. Supported markers are **bold**, _italic_, ~~strikethrough~~, `monospace`, and ||spoiler||. Markers are removed, nesting is supported, a backslash escapes the following character, and unmatched markers remain literal. Without the option, the body is sent exactly as provided.
  • Request body limit: 64 KiB.
  • No other fields are accepted. Every POST creates a fresh message ID; repeating the same request can send a duplicate.
  • If Signal explicitly rejects the encrypted payload because the recipient device list changed, the daemon preserves unaffected sessions, selectively repairs missing, extra, stale, or registration-changed devices, and retransmits once inside the same POST. This bounded retry is safe because Signal rejected the first transmission before accepting it. A second mismatch returns retryable HTTP 502.
  • Other transport failures are not retried automatically. If the connection closes before the response, the outcome is ambiguous.

React to a message

Call authenticated POST /v1/reactions with the same destination, the webhook's opaque message.message_id, and one supported emoji: 

curl --fail-with-body \
  -H "Authorization: Bearer ${SIGNAL_API_TOKEN}" \
  -H 'Content-Type: application/json' \
  --data '{
    "destination": "+12025550123",
    "message_id": "incoming-message-id",
    "emoji": "👍"
  }' \
  http://127.0.0.1:8080/v1/reactions

The target must exist locally and belong to the destination conversation. Sending another reaction from this account replaces its locally stored reaction after Signal accepts the new reaction. Reaction removal is not currently exposed by the API.

Call it from another Docker Compose project

Attach both projects to one private external network. Create it once, then add this override beside compose.yaml: 

docker network create app-private
# compose.override.yaml
services:
  signal:
    networks:
      default: {}
      app-private:
        aliases: [signal-cli]

networks:
  app-private:
    external: true

Attach your application service to the same app-private network. It can then call http://signal-cli:8080/v1/messages with the same bearer token. A webhook service on that network can likewise use an internal URL such as http://your-app:3000/signal/webhook. Keep the network private and retain API authentication.

7. Receive incoming webhooks

The daemon calls the configured SIGNAL_WEBHOOK_URL in two ways. Your endpoint must support both methods when webhooks are enabled:

Purpose Method Request Successful response
Startup check GET No body and no webhook signature. Exactly HTTP 200; the response body is ignored.
Message delivery POST JSON body shown below. When SIGNAL_WEBHOOK_SECRET is set, includes X-Signal-Webhook-Signature. Any HTTP 2xx; the response body is ignored.

For example, a delivery consumer can acknowledge an accepted update without a body: 

HTTP/1.1 204 No Content

Set SIGNAL_WEBHOOK_URL before starting the daemon. During every startup, before Signal and the control API become ready, the daemon sends a bodyless GET to that exact URL. Only an exact HTTP 200 succeeds. Redirects are not followed; timeouts, network errors, and all other statuses fail startup. The GET has no webhook HMAC signature. With Compose's restart policy, Docker keeps restarting the daemon until the endpoint returns 200. No check runs when the URL is unset.

After the startup check succeeds, each newly persisted supported direct text becomes one JSON POST: 

{
  "webhook_update_id": "7484932459384721",
  "message": {
    "message_id": "incoming-message-uuid",
    "date": 1783960000000,
    "text": "hello",
    "from": { "id": "sender-aci" },
    "chat": { "id": "sender-aci", "type": "private" },
    "reply_to_message": {
      "message_id": "1783959000000",
      "date": 1783959000000,
      "text": "quoted text",
      "from": { "id": "quoted-author-aci" },
      "chat": { "id": "sender-aci", "type": "private" }
    }
  }
}

IDs are strings. date is Signal's exact sent time in Unix milliseconds, matching Signal Desktop. Because only direct messages are supported, chat.id and from.id are both the sender's stable Signal ACI; that ACI can also be used as the send API's destination. webhook_update_id is separately derived from message_id, allowing consumers to deduplicate retries without relying on timestamps, which are not guaranteed to be unique. Queued outbox entries that still use the former update_id name are migrated automatically when loaded.

For a quoted reply, reply_to_message contains Signal's quoted author, text, and target sent timestamp. Signal identifies a quote target by author and sent timestamp rather than by this daemon's local message UUID, so the nested message_id is the target timestamp encoded as a string and date is the same value as a number. The field is omitted for messages without a quote. Quoted attachments are not yet supported by the headless receiver. Formatting on quoted text is ignored so that the reply itself is still delivered.

Verify webhook signatures

When SIGNAL_WEBHOOK_SECRET is set, requests include: 

X-Signal-Webhook-Signature: sha256=HEX_HMAC_SHA256

Compute HMAC-SHA256 over the exact raw HTTP body and compare it in constant time before parsing or acting on the request.

Delivery contract

  • Ordered, at-least-once delivery.
  • Only HTTP 2xx removes the oldest durable update. The daemon then marks the message read locally, sends a read sync to the account's other linked devices and, when the restored profile's read-receipt setting is enabled and the contact's message request has been accepted, sends a Signal read receipt to the contact. Network errors, redirects, timeouts, and non-2xx responses keep the update queued and leave the message unread.
  • Post-webhook read actions run separately with the same bounded timeout. A failure is reported but never blocks durable enqueue, delays acknowledgement of later Signal envelopes, or redelivers an already-successful webhook. A process exit after durable webhook removal can leave the read actions incomplete.
  • Failures retry from one second up to five minutes.
  • Default attempt timeout: ten seconds.
  • The encrypted outbox lives inside the profile and moves with R2 snapshots.
  • Consumers must deduplicate by stable string webhook_update_id.
  • SIGNAL_WEBHOOK_MAX_PENDING limits durable, undelivered webhook updates, not messages in Signal's database. At the limit, receive acknowledgement and startup reconciliation pause until delivery frees space; queued updates are not evicted.
On first initialization, historical linked messages are baselined and not delivered as a webhook flood. If no URL is configured, the cursor advances without building a pending queue.

8. Switch between headless and GUI modes

Stop one owner fully before starting the other.

Headless → GUI

docker compose stop --timeout 45 signal
docker compose --profile ui up signal-ui

GUI → headless

docker compose --profile ui stop --timeout 150 signal-ui
docker compose up -d signal

If the profile lock reports that it is busy, find and stop the actual owner. Never delete or bypass the lock while a container may still be using the profile.

9. Routine operations

Logs and status

docker compose ps
docker compose logs --tail=200 signal
curl --fail http://127.0.0.1:8080/healthz
curl --fail-with-body http://127.0.0.1:8080/readyz

Offline profile validation

SIGNAL_DAEMON_CONNECT=false docker compose up signal

Offline mode opens and validates the profile without connecting to Signal. The API remains local, but message sends cannot succeed.

Later machine migration

This flow is bidirectional, including remote back to the original computer. Whichever stopped machine currently owns the live profile is the source; the stopped machine taking ownership is the destination.

  1. Stop the current owner and wait for exit.
  2. Push and record a snapshot UUID.
  3. Verify that exact UUID.
  4. Stop the destination owner.
  5. Pull the UUID with --replace.
  6. Start exactly one destination runtime.

Rollback

docker compose stop --timeout 45 signal
docker compose --profile tools run --rm state verify OLD_SNAPSHOT_UUID
docker compose --profile tools run --rm state pull OLD_SNAPSHOT_UUID --replace
docker compose up -d signal
Rollback discards local messages and protocol state newer than the selected snapshot. Signal may not replay them.

10. Update from upstream

Builds expire exactly 90 days after image build time. Signal may also remotely shorten GUI validity. The official fork policy is a 90-day build lifetime; merge current upstream and rebuild at least that often.

Configure the official repository once, then update by fetching, merging, resolving conflicts, and rebuilding. No timestamp, revision, expiration, Compose, or .env bookkeeping is required. 

git remote add upstream https://github.com/signalapp/Signal-Desktop.git # once
git fetch upstream
git merge upstream/main
# Resolve conflicts, if any, and complete the merge.

docker compose build signal signal-ui state
docker compose up -d signal
curl --fail http://127.0.0.1:8080/readyz

Docker automatically generates one build timestamp shared by the GUI and daemon. There is no timestamp environment variable or build argument to provide. The separate reproducible-build script derives its timestamp from the latest Git commit and fails instead of guessing when Git metadata is unavailable.

Published daemon image

Every push to main automatically runs the project linters plus the daemon and state-tool tests, builds and smoke-tests the Linux amd64 daemon target, and publishes it to GitHub Container Registry as ghcr.io/vladtsap/signal-desktop-cli only if every preceding step succeeds. The tested local image is pushed without rebuilding it. The workflow publishes both the moving latest tag and an immutable tag containing the full commit SHA. 

docker pull ghcr.io/vladtsap/signal-desktop-cli:latest
docker pull ghcr.io/vladtsap/signal-desktop-cli:FULL_COMMIT_SHA

The publish job exposes image (the immutable name@sha256:digest reference), image_name, latest_tag, commit_tag, and digest outputs for later jobs, and writes the reusable references to the workflow summary. The package contains only signal-daemon; build the UI and state-transfer tool locally through Compose. GitHub controls package visibility separately from repository visibility. Private packages require a GitHub token with read:packages permission when authenticating Docker to ghcr.io.

11. Troubleshooting

/readyz returns 503
Read the JSON phase and reason. Common causes are an unlinked profile, Signal transport reconnecting, a terminal transport failure, an expired build, or missing profile credentials. Check docker compose logs signal.
The profile lock is busy
Run docker compose ps and stop the UI, daemon, or state tool that owns the profile. Do not remove the lock file to force concurrent access.
An E164 destination is not found
E164 works only if the restored Signal profile already maps that number to an ACI. Use a lowercase ACI directly or first establish the contact mapping through supported Signal Desktop activity.
Webhook deliveries repeat
This is expected under at-least-once delivery when the consumer accepts a request but the daemon does not durably observe its 2xx response. Deduplicate using webhook_update_id.
Readiness is green but incoming webhooks stop
The authenticated transport probes /v1/keepalive every 30 seconds and reconnects a socket that times out or returns a non-2xx response. Webhook POSTs and post-webhook read actions do not hold the durable enqueue queue; a slow endpoint or read receipt therefore cannot prevent acknowledgement of later Signal envelopes. Check docker compose logs signal for HeadlessMessageReceiver: failed to process incoming Signal envelope. Failed envelopes are rejected to Signal for retry rather than acknowledged; investigate the logged decrypt or persistence error. Recipient device-list mismatches during an API send are repaired selectively and do not reset unaffected inbound sessions.
R2 restore refuses a non-empty profile
Verify the intended snapshot and rerun pull with --replace. Ensure all profile owners are stopped first.
The daemon says the profile is unlinked
Restore a known-good linked snapshot or link a fresh profile with the GUI. Do not manually edit config.json, SQLCipher, identities, sessions, or prekeys.

Daemon, API, and webhook

Variable Default Purpose

SIGNAL_API_TOKEN unset Bearer token, at least 16 characters; required online.
SIGNAL_DAEMON_CONNECT true Enable or disable Signal network connection.
SIGNAL_DAEMON_LOG_LEVEL info debug, info, warn, or error.

SIGNAL_WEBHOOK_URL unset HTTP(S) endpoint; exact GET 200 required during startup.
SIGNAL_WEBHOOK_SECRET unset HMAC secret, at least 16 characters.
SIGNAL_WEBHOOK_TIMEOUT_MS 10000 Startup GET and delivery POST timeout.
SIGNAL_WEBHOOK_MAX_PENDING 1000 Maximum durable undelivered updates; a full queue applies receive backpressure.
SIGNAL_MEMORY_LIMIT 768m Hard daemon Compose memory limit.
SIGNAL_PIDS_LIMIT 128 Daemon process limit.

When SENTRY_DSN is set, the daemon reports uncaught exceptions, unhandled promise rejections, startup and shutdown failures, and handled background failures from webhook delivery/read actions, Signal receive and transport processing, and pre-key maintenance. Events include the daemon release and an operation tag. API requests, HTTP headers, webhook payloads, Signal message content, and HTTP breadcrumbs are deliberately excluded. Sentry is disabled when the variable is unset. Retryable failures may produce an event for each failed attempt, subject to Sentry's own rate limits.

If the daemon exceeds SIGNAL_MEMORY_LIMIT, its cgroup OOM-kills the container, commonly with exit code 137. Compose then restarts it because the service uses restart: unless-stopped. Durable database and webhook state recover, but an interrupted send has an ambiguous outcome and no deduplication key for safe replay. Raise the limit or remove mem_limit in a private Compose override if normal workloads repeatedly reach it.
